# Ralph Engine — Ledger Service TDD

## Project Context

- **Product:** Build an Organization-scoped, domain-neutral double-entry Ledger service for authorized Service Actors, Operators, finance users, and auditors. State-changing intent enters through one durable asynchronous typed Command Task ingress; authoritative Domain Events and append-only Journal History record committed facts; rebuildable CQRS read models expose balances, activity, reconciliation, command status, Controls, and the Audit Trail. This is not a cryptocurrency ledger, product model, payment-settlement engine, or consumer application.
- **Source authority:** Follow `CONTEXT-MAP.md`, `SHARED-LANGUAGE.md`, each `contexts/*/CONTEXT.md`, and accepted ADRs for canonical language, ownership, and decisions. Use `specs/001-ledger-service/spec.md` for feature behavior. Treat model notes as derived aids and `specs/001-ledger-service/contracts/openapi.json` as non-authoritative supporting evidence whose documented drift must be reconciled by an explicit work item before implementation against that contract.
- **Critical behaviors under test:**
  - Keep every Organization and Ledger isolated. A Transaction stays within one Ledger, contains at least two strictly positive Postings, has a non-zero Net Account Effect for at least one Account, and balances Debits and Credits independently for every Asset.
  - Accept a complete Transaction and all of its Postings atomically at one Ledger Position only after every affected Account Revision and applicable Control decision remains valid. Any validation, concurrency, or durability failure commits none of the change set.
  - Preserve immutable accepted Transactions, Postings, Domain Events, Control history, audit evidence, and consumed-key bindings. Corrections append linked Reversal or Adjustment Transactions; Pending content changes use non-branching void-and-replace links; resolutions append complete settlement and optional Continuation Transactions atomically.
  - Enforce the staged Account lifecycle and its opening and closure gates. Balance Floor and Balance Ceiling changes are prospective; temporary prohibitions remain Controls rather than Account States.
  - Admit mutations only as durable asynchronous typed Command Tasks. Server policy selects immediate or approval-required execution, expected revisions are checked at execution, and terminal domain results are obtained through Task status rather than returned inline from submission.
  - Make idempotency exact across every mutating operation and Ledger in one Organization-and-client namespace. Same key plus canonical content returns the original outcome; different content conflicts; retries never duplicate Tasks, Transactions, state changes, or Limit Usage. Source Reference is correlation, never retry identity.
  - Evaluate all Active applicable Controls and consume every applicable Usage Limit in the same authoritative decision as the controlled action. Derived balances and capacities inform callers but never authorize a mutation.
  - Derive current and historical balances from authoritative history. Posted Balance uses the Account's Normal Side; Point-in-Time Balance uses Effective At plus at most one of Known At or Known Through and excludes pending totals and capacities. Rebuilding a view at the same history position produces the same result.
  - Record exactly one Action Record for each externally visible attempt and system execution attempt, including reads, rejections, failures, and replays. Only committed subject transitions produce causally linked State Change Records; audit payloads exclude secrets and unapproved sensitive values.
  - Preserve Recorded At, Effective At, Ledger Position, and Consistency Position as distinct concepts. Read-after-write queries either meet the requested consistency position or report a clear not-yet-current result.
- **Build requirements:**
  - The repository currently contains specifications and tested shell tooling, but no .NET solution, application project, or product tests. The executable gates are `bash .specify/scripts/bash/tests/test-list-specs.sh` and shell syntax validation. Do not invent missing protocol, persistence, aggregate, event-stream, or deployment decisions inside a feature work item.
  - .NET 10 is installed for the intended application. Once an explicitly scoped scaffolding work item creates the repository solution and test projects, update `hooks.yaml` to name that solution, restore when its dependency graph changes, build before any `dotnet test --no-build` gate, and require both the full test suite and `dotnet build --no-restore` to pass.
  - Test externally observable behavior through the public seam named and confirmed in the current work item. If the work item does not name a seam, or relies on the unreconciled OpenAPI contract, stop and request that decision before writing a test.
  - Use a real test database where practical. Mock only nondeterministic or external system boundaries such as time, randomness, external APIs, or filesystem access; do not mock internal bounded-context collaborators merely to assert calls.
  - Tests must be deterministic, use independent worked examples or literals for expected accounting results, and leave no network, wall-clock, ordering, or shared-state dependency uncontrolled. Do not add skipped tests or weaken assertions to make a gate pass.
- **Refactor boundaries:**
  - Keep Accounts, Journal, Balances, Products, Customers, Payment Instruments, and Controls as distinct language and ownership boundaries even when an initial ACID transaction co-locates Accounts and Journal writes. Do not move product concepts into accounting contexts or let Balances projections authorize writes.
  - Preserve the authoritative event history, Journal atomicity, Account Revision Fence, audit causality, idempotency semantics, replay determinism, and rebuild equivalence. A refactor must not rewrite accepted history, change an event contract in place, or turn a derived view into a second source of truth.
  - Do not choose unresolved context ownership, public contract shape, storage schema, aggregate boundary, event-stream topology, Controls coordination protocol, or deployment boundary as incidental cleanup. Such changes require their own explicit work item and, where architectural, an accepted ADR.
  - Keep tests at stable public seams and assert outcomes rather than internal calls, private methods, database rows, or implementation structure. Refactoring belongs to a separately scoped review step after the behavior cycle is green, not inside red-to-green work.
  - Change only the current work item's behavior and the minimum supporting code. Preserve unrelated user changes and leave the entire tree green before ending the session.

## Runtime Rules

- Read the current work item and the canonical context documents and ADRs that govern it before changing code. If sources conflict, follow the authority order above and report the drift rather than silently choosing.
- Start every behavior change with one test at the confirmed public seam. Run it and verify that it fails for the intended missing behavior, not because of compilation, setup, fixture, or infrastructure failure.
- Implement only enough production code to make that one test pass. Do not batch imagined future tests, speculative abstractions, or multiple behaviors into a cycle.
- Keep runtime changes behind typed contracts. Update tests, validation gates, release expectations, and policy wording together whenever their contract changes; never allow them to drift silently.
- Keep each step small enough to understand and review. Do not skip tests, mute failures, loosen assertions, or inspect private state to make progress look faster.
- Once the focused test is green, run all required tests and the build. End the behavior cycle there; perform refactoring only as separately scoped green-tree work.
- Process at most the one work item selected for this session and leave the entire tree green before reporting it complete.
